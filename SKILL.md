# tech-domain-learning

面向编程或任意技术领域的系统化学习。生成学习路线 ROADMAP.md、进度追踪 PROGRESS.md，按阶段交付理论讲义 THEORY.md 与可运行示例 demo/。

## 触发短语

帮我学 X · 制定 X 学习路线 · 生成阶段讲义 · 下一阶段 · 本阶段学完了 · 我有疑问

## 何时调用

| 用户表述 | 对应工作流 |
|----------|-----------|
| 帮我学 / 系统学习 / 深入学习 X | 工作流 A |
| 制定 X 学习路线 / 路线图 | 工作流 A |
| 我想成为 X 架构师 / 工程师，该怎么学 | 工作流 A（含前置知识自检） |
| 生成阶段 N 的讲义 | 工作流 B |
| 下一阶段 / 继续 | 工作流 B（取第一个未完成阶段） |
| 我对这部分有疑问 / 补充一下 | 工作流 C |
| 本阶段学完了 / 验收通过 | 工作流 D |

**不触发**：简单查阅（如"这个函数怎么用"）、非学习意图（如"帮我写个脚本"）、已明确指定其他技能的场景。不确定时直接问用户是否需要建立学习路线。

---

## 核心原则

1. **重内容，轻形式**：讲清楚比模板填满重要。节数、小节数由主题复杂度决定，不设下限。
2. **知识可验证**：URL 与版本号经 WebSearch 确认；引用优先官方文档与权威出版物。
3. **单一进度来源**：`PROGRESS.md` 是唯一进度真相。阶段文件夹名与路线中的阶段名严格一致。
4. **先审阅再生成**：交付 ROADMAP.md 后暂停，用户确认前不生成阶段内容。

## 目录结构

```
learning/<技术专题简称>/
├── ROADMAP.md
├── PROGRESS.md
└── <序号>-<阶段名>/
    ├── THEORY.md
    └── demo/
        └── README.md
```

`learning/` 相对于本 skills 仓库根目录生成。THEORY.md 参考 [`reference/theory.md`](reference/theory.md)，demo/README.md 参考 [`reference/demo-readme.md`](reference/demo-readme.md)。

## 工作流 A：初始化路线

1. 确认技术方向，**WebSearch** 检索当前生态：主流版本、官方文档入口、核心工具链
2. 编写 `ROADMAP.md`：通常 3–8 阶段 + ASCII 箭头图 + 阶段总览表 + 各阶段详情（模板见 [`reference/roadmap-progress.md`](reference/roadmap-progress.md)）。窄专题（如单个协议/算法）可少至 2–3 阶段，宽专题（如全栈架构）可扩至 6–8
3. 初始化 `PROGRESS.md`：阶段清单与路线一致，当前阶段标「未开始」
4. **暂停，请用户审阅路线**——确认前不生成阶段 1 内容
5. 用户确认后 → 进入工作流 B

## 工作流 B：生成阶段内容

1. 创建阶段文件夹 `learning/<专题>/<序号>-<阶段名>/`
2. 撰写 `THEORY.md`：讲清核心概念、原理、实践要点，善用 Mermaid 图与对比表格辅助理解，附带经 WebSearch 验证的推荐阅读链接
3. 创建 `demo/`：最小可运行示例 + `README.md`（必含：总览表、环境要求、运行命令、学习建议）。示例形式随技术方向适配——SQL 脚本、Docker Compose 多服务、配置文件集、单项目代码均可
4. 更新 `PROGRESS.md` 中本阶段状态为「进行中」

## 工作流 C：学习中的问答补充

在 THEORY.md 或 demo/ 中追加内容。**不覆盖用户手写笔记**。若新增文件或改变学习顺序，同步更新 demo/README.md。

## 工作流 D：阶段完成

1. `PROGRESS.md` 阶段清单：状态 → 已完成，填写日期与时长
2. 完成记录追加一条：日期、时长、2–5 条简要收获
3. 推进当前阶段到下一阶段
4. 提示用户下一阶段文件夹名与入口文件

## 质量自检

| # | 检查项 |
|---|--------|
| 1 | 路线图含生成日期与「以官方文档为准」声明 |
| 2 | URL 经 WebSearch 验证，引用官方文档或权威出版物 |
| 3 | 讲清是什么、为什么——简单主题一两段说清即可，复杂概念才展开三层论证；善用 Mermaid 图表和对比表格辅助理解 |
| 4 | 每节末尾有核心概念总结 + 拓展提问提示词，方便读者回顾和延伸学习 |
| 5 | 段落读起来像教学，不像填表 |

## 常见问题

**Q：用户想跳过某些阶段？**
按用户要求调整 `ROADMAP.md` 和 `PROGRESS.md`，保持一致。

**Q：用户说"我已经会了阶段 1–2，从 3 开始"？**
将阶段 1–2 标记为已完成（备注「用户自评已掌握」），当前阶段设为阶段 3。

**Q：用户对路线有异议？**
讨论调整方案，达成一致后更新 `ROADMAP.md` 和 `PROGRESS.md`。

**Q：`demo/` 中放什么类型的代码？**
取决于技术方向：Java 用 Maven 项目；前端用组件文件；消息队列用 Docker Compose + 客户端示例；数据库用 SQL 脚本。原则：**最小化、可运行、可修改做实验**。

## 异常处理

| 场景 | 处理方式 |
|------|----------|
| WebSearch 不可用或 URL 无法验证 | 标注`⚠ 待验证`，引导用户自行确认 |
| 用户提供的技术方向模糊或过于宽泛 | 追问 1–2 个澄清问题后再生成路线 |
| `learning/<专题>/` 目录已存在 | 先读已有 ROADMAP.md 与 PROGRESS.md，按当前进度继续，不覆盖 |
| 用户提供的版本号与当前生态明显不符 | 提示差异，以 WebSearch 结果为准，并标注原因 |
