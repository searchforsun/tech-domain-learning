# THEORY.md 写作指引

理论讲义的目标：读者读完一节后能**用自己的话说出核心概念，并能动手验证**。

## 教学节奏（推荐）

每节的展开遵循「定位 → 论证 → 落地」的自然节奏：

| 层次 | 做什么 | 怎么写 |
|------|--------|--------|
| **定位** | 建立认知锚点 | 首段说清「是什么 + 为什么重要」，给出结论取向；配上知识点索引表或分条 |
| **论证** | 讲透因果与边界 | 用完整因果句展开原理；带反例或易混点对比；列表前后各有一句总起/收束 |
| **落地** | 具象化 + 可验证 | 嵌入短例/对比表/Mermaid 图/代码片段；长脚本放 `demo/`；指名对应 demo 路径 |

这不是硬性模板——简单主题一两段就能说清的不必拆分三层。写到复杂概念时用它自查：读者看完知道是什么吗？知道为什么吗？能动手验证吗？

## 结构原则

1. **节数由主题决定**：与 ROADMAP.md 核心主题数对应。不套固定骨架。
2. **每节讲清一件事**：先讲是什么、为什么重要，再讲怎么用，给具体例子。有因果论证，不只列条目。
3. **避免**：整节只有表格/列表无论证；堆砌术语不解释；长篇复制官方文档。

## 知识地图（主题数 ≥ 4 时使用）

文件头部之后放一个导航表，让读者一眼看到全局：

```markdown
## 本阶段知识地图

| 块 | 你要带走的抓手 |
|----|----------------|
| 一 | <核心主题 A> |
| 二 | <核心主题 B> |
| 三 | <核心主题 C> |
```

紧接地图之后，加路线要点与章节的映射——确保 ROADMAP.md 的每条要点都有正文展开，读者也能快速定位：

```markdown
**路线要点 ↔ 本文章节**

| `ROADMAP.md` 阶段 N 要点 | 本文展开位置 |
|--------------------------|----------------|
| <要点 1> | **一** |
| <要点 2> | **二** |
| <要点 3> | **三** |
```

## 可视化与结构化表达

善用图表把抽象概念具象化，这是理论讲义的亮点手段：

- **Mermaid 图**：适合流程、时序、状态机、组件关系。图前一句说明目的，图后 1–2 句收束。
- **对比表格**：适合概念辨析、方案权衡、场景选型。表格前后各有一句过渡，不孤立出现。
- **代码短例**：正文嵌入 ≤10 行关键片段，长脚本放 `demo/`。

不作为硬性指标，但写完一节回头看：全是文字墙吗？是的话很可能需要一个图或表。

## 每节收尾：核心概念 + 拓展提问

每节末尾加一段提要，方便读者回顾要点和延伸学习：

```markdown
---

**本节提要**

- **核心概念**：<3–5 个关键短语，分号分隔>
- **拓展提问提示词**

> 主题：<本节完整主题>。核心概念：<列出所有核心概念>。请拓展：<具体想深入的方向，不依赖上下文，复制即可用>
```

**要点**：
- 核心概念是本节最关键的 3–5 个术语/结论，读者一眼看到要带走什么
- 拓展提问提示词是给读者的跳板——复制粘贴到 AI 对话即可继续深挖，不依赖上文

## 文件头部

```markdown
# 阶段 N：<阶段标题>

> 与 ROADMAP.md 阶段 N 对齐。示例见 demo/README.md。
> 推荐阅读链接基于撰写时检索，请以官方文档为准。

（1–2 句：本阶段在路线中的位置与核心目标）
```

## 推荐阅读

文末列出推荐阅读，每条包含：**标题**、关联主题、完整 URL（经 WebSearch 验证）、检索关键词。URL 单独一行，不在同行拼接其他字符。

```markdown
## 推荐阅读

- **文档标题**
  - 关联主题：……
  - https://完整URL
  - 检索：`关键词`
```
